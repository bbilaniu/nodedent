# Interactive Endodontic Chairside Decision Guide + Note Generator Proposal

## Purpose

Create a web app that transforms a detailed endodontic treatment algorithm into a chairside decision guide. The app should guide the clinician through treatment one decision at a time, while automatically building a structured clinical note from the decisions, measurements, and procedural events recorded during the appointment.

The goal is **not** to display a long 99-step checklist. The goal is to create a clinical state machine that shows the current safest action, captures the required clinical data, branches appropriately when difficulty increases, and generates a useful endodontic note.

---

## Core Product Idea

The app should combine four layers:

```text
Protocol nodes
   ↓
Decision engine
   ↓
Chairside UI
   ↓
Structured note generator
```

Each clinical step should be represented as a protocol node. Each node should define:

- The current phase
- The chairside instruction
- Required instruments or materials
- Required inputs
- Available decision options
- The next node after each decision
- The note event generated by that decision
- Any difficulty or referral flag triggered by that decision

The clinician should only see the current action and the relevant decision choices. The app should quietly accumulate structured data in the background.

---

## Design Principle

Every chairside click should do at least one of the following:

1. Advance the clinical state.
2. Record a clinically meaningful fact.
3. Trigger a safety, difficulty, medication, temporization, or referral pathway.
4. Add a structured event to the clinical note.

Avoid generic buttons such as:

```text
Next
Continue
Done
```

Prefer clinically meaningful buttons such as:

```text
10C reached estimated working length
10C stopped short
EAL 0 established
Guide path file reached EAL 0
Final shape achieved
Paper point remains wet
Medicate and temporize
Refer
```

These labels become the backbone of the generated clinical note.

---

## Major Phases

The endodontic workflow should be grouped into phases rather than displayed as a linear 99-step list.

Recommended phases:

1. Pre-op setup
2. Access
3. Initial canal scouting
4. Working length
5. Glide path
6. Shaping
7. Smear layer removal
8. Final disinfection
9. Obturation gauging
10. Gutta-percha cone fit
11. Drying
12. Sealer placement
13. Cone seating
14. Downpack / searing / backfill
15. Closure
16. Medication, temporization, and referral pathway

Each phase should contain small decision cards.

## Canal-Based Data Model

The app should be tooth-based globally but canal-based after access.

Example case structure:

```ts
type EndoCase = {
  tooth: string;
  procedureType: "RCT" | "RCT initiated" | "Retreatment" | "Emergency pulpectomy";
  diagnosis?: {
    pulpal?: string;
    apical?: string;
  };
  preOp?: {
    radiographsReviewed?: boolean;
    cbctReviewed?: boolean;
    estimatedChamberDepth?: number;
  };
  canals: CanalRecord[];
  globalEvents: ClinicalEvent[];
  closure?: ClosureRecord;
};

type CanalRecord = {
  name: string; // MB, MB2, DB, P, ML, DL, etc.
  estimatedWorkingLength?: number;
  availableTreatmentSpace?: number;
  referencePoint?: string;
  eal0?: number;
  patencyLength?: number;
  shapingLength?: number;
  finalShape?: string; // e.g. 30/.04
  obturationGauge?: string;
  masterCone?: string; // e.g. 30/.04 GP
  coneFitRadiograph?: "acceptable" | "short" | "long" | "not taken";
  dryingStatus?: "dry" | "slightly damp" | "wet" | "persistent wet";
  events: ClinicalEvent[];
};

type ClinicalEvent = {
  id: string;
  timestamp: string;
  type: string;
  tooth?: string;
  canal?: string;
  details?: Record<string, unknown>;
};

type ClosureRecord = {
  type: "temporary" | "orifice barrier and temporary" | "final restoration";
  material?: string;
  notes?: string;
};
```

## Protocol Node Model

The protocol should be data-driven, not hardcoded into React components.

```ts
type DifficultyFlag = "none" | "caution" | "high" | "refer";

type DecisionOption = {
  label: string;
  nextNodeId: string;
  noteEvent?: {
    type: string;
    details?: Record<string, unknown>;
  };
  difficultyFlag?: DifficultyFlag;
};

type ProtocolNode = {
  id: string;
  phase: string;
  title: string;
  chairsideInstruction: string;
  instruments?: string[];
  materials?: string[];
  requiredInputs?: string[];
  safetyNotes?: string[];
  options: DecisionOption[];
};
```

Example node:

```ts
const advance10CToEstimatedWL: ProtocolNode = {
  id: "advance-10c-to-estimated-wl",
  phase: "Initial canal scouting",
  title: "Advance 10C hand file",
  chairsideInstruction:
    "Place lubricant or aqueous irrigation. Advance 10C hand file to passive resistance. Do not exceed estimated working length.",
  instruments: ["10C hand file"],
  materials: ["file lubricant or aqueous irrigation"],
  requiredInputs: ["canal", "estimatedWorkingLength", "fileTerminalLength"],
  safetyNotes: ["Do not exceed estimated working length."],
  options: [
    {
      label: "10C reached estimated working length",
      nextNodeId: "open-canal-orifice",
      noteEvent: {
        type: "scouting.estimatedWorkingLengthReached"
      }
    },
    {
      label: "10C stopped short",
      nextNodeId: "measure-available-treatment-space",
      noteEvent: {
        type: "scouting.fileStoppedShort"
      }
    }
  ]
};
```

## Decision Engine

The decision engine should be a pure function. React should render the current node and call the engine when a decision is selected.

```ts
type DecisionEngineInput = {
  currentNodeId: string;
  selectedOptionLabel: string;
  caseData: EndoCase;
};

type DecisionEngineOutput = {
  nextNodeId: string;
  updatedCaseData: EndoCase;
  generatedEvent?: ClinicalEvent;
  warnings?: string[];
};

function applyDecision(input: DecisionEngineInput): DecisionEngineOutput {
  // 1. Find current protocol node
  // 2. Find selected decision option
  // 3. Generate event if specified
  // 4. Update case data
  // 5. Apply difficulty flag if needed
  // 6. Return next node
}
```

Keep all clinical branch logic out of JSX wherever possible.

## Reusable Troubleshooting Modules

The source algorithm contains repeated loops. These should become reusable modules instead of duplicated UI.

### Regain Patency Module

Used whenever the protocol requires a return to patency work.

```text
Regain Patency

1. Irrigate.
2. Place lubricant or aqueous irrigation.
3. Measure 10C hand file to patency length.
4. Advance to reference point.
5. Work in filing / reciprocating / watch-winding motion.
6. Continue until 10C is super loose.
7. Reassess.

Outcomes:
[Patency regained]
[Still short]
[File stop]
[File resistance]
[Medicate and temporize]
[Refer]
```

### File Stop vs File Resistance Module

```text
Determine whether the file is encountering resistance or a stop.

Outcomes:
[File resistance]
[File stop]
[Uncertain]
[Difficulty exceeds comfort]
```

### Persistent Wet Canal Module

```text
Persistent wet canal

1. Repeat drying with measured paper points.
2. If still wet after several attempts, place calcium hydroxide.
3. Close access with sponge and temporary restorative material.
4. Resume at smear layer removal / disinfection pathway next visit if appropriate.

Outcomes:
[Canal dried]
[Persistent wet canal]
[Calcium hydroxide placed]
[Temporized]
[Refer]
```

### Gutta-Percha Cone Adjustment Module

```text
Cone fit issue

If cone is short:
- Try one size smaller.
- If still short, return to shaping.

If cone is long:
- Crimp at reference point.
- Measure engaged cone.
- Subtract shaping length.
- Trim excess.
- Retry cone fit.

Outcomes:
[Cone fit acceptable]
[Cone short]
[Cone long]
[Return to shaping]
```

## Difficulty Flags and Stop Rules

The app should explicitly surface difficulty changes.

Suggested difficulty levels:

```text
Green: routine pathway
Yellow: increased difficulty / proceed with caution
Red: high difficulty / consider medication, temporization, or referral
Black: stop pathway / referral or staged treatment strongly recommended
```

Important trigger examples:

```text
Available treatment space ≤16 mm
File remains >3 mm short of estimated working length
File stop cannot be bypassed
Repeated failure to regain patency
Guide path file cannot reach guide path length
Final shape cannot be created predictably
Cone repeatedly cannot fit to shaping length
Canal remains wet after repeated drying
Clinician discomfort or difficulty exceeds skill level
```

When a trigger occurs, show a banner:

```text
Warning: available treatment space is ≤16 mm.
Case difficulty increases significantly.

Options:
[Proceed with extreme caution]
[Medicate and temporize]
[Refer]
[Document difficulty and continue]
```

The note should automatically include the selected response.

Example note fragment:

```text
Available treatment space was limited to ___ mm. Increased case difficulty was noted. Treatment was continued with caution / canal was medicated and temporized / referral was recommended.
```

## Note Generator Strategy

The note generator should be event-based.

Do not generate the note directly from the current UI screen. Generate it from the case data and the event log.

Example event log:

```ts
const events: ClinicalEvent[] = [
  {
    id: "evt_001",
    timestamp: "2026-05-26T10:15:00",
    type: "access.completed",
    tooth: "36",
    details: {
      chamberAccessConfirmed: true,
      canalsLocated: ["MB", "ML", "DB", "DL"]
    }
  },
  {
    id: "evt_002",
    timestamp: "2026-05-26T10:28:00",
    type: "workingLength.established",
    tooth: "36",
    canal: "MB",
    details: {
      eal0: 20.5,
      patencyLength: 21.5,
      shapingLength: 19.5,
      referencePoint: "MB cusp"
    }
  },
  {
    id: "evt_003",
    timestamp: "2026-05-26T10:45:00",
    type: "shaping.completed",
    tooth: "36",
    canal: "MB",
    details: {
      finalShape: "30/.04"
    }
  }
];
```

Example note fragment map:

```ts
const noteFragments = {
  "access.completed": (event: ClinicalEvent) => {
    const canals = event.details?.canalsLocated as string[] | undefined;
    return `Access completed. Pulp chamber entered and refined. Canals located: ${canals?.join(", ")}.`;
  },

  "workingLength.established": (event: ClinicalEvent) => {
    const d = event.details as {
      eal0: number;
      patencyLength: number;
      shapingLength: number;
      referencePoint: string;
    };

    return `${event.canal}: EAL 0 ${d.eal0} mm, patency length ${d.patencyLength} mm, shaping length ${d.shapingLength} mm, reference point ${d.referencePoint}.`;
  },

  "shaping.completed": (event: ClinicalEvent) => {
    const d = event.details as { finalShape: string };
    return `${event.canal}: final shape ${d.finalShape}.`;
  }
};
```

```text
ML: ...
DB: ...
DL: ...

Canals irrigated with NaOCl during instrumentation. Smear layer removed with 17% EDTA. Final disinfection completed with NaOCl.

Final shapes:
MB 30/.04
ML 30/.04
DB 30/.04
DL 30/.04

Master cones fitted and confirmed radiographically. Bioceramic sealer placed. Gutta-percha cones seated. Downpack/backfill completed. Chamber cleaned. Access restored with temporary restorative material.
```

### 2. Compact EMR Note

```text
36 RCT. RD isolation. Access completed, canals MB/ML/DB/DL located. WLs established with EAL/PA. NaOCl irrigation throughout, 17% EDTA smear layer removal, final NaOCl. Shaped to 30/.04. GP cone fit confirmed PA. Bioceramic sealer + GP obturation completed. Access temporized. POIG.
```

## Implementation Strategy

### Stage 1: Build the generic engine

Create a generic decision engine that can later support other chairside guides.

```text
Input:
- current node
- selected option
- current case data

Output:
- next node
- updated case data
- generated event
- warnings
```

### Stage 2: Encode the endodontic MVP

Enter only the most important nodes first. Do not encode all 99 steps immediately.

### Stage 3: Build the note generator

Generate note fragments from events and canal records.

### Stage 4: Add the full protocol

Convert remaining steps into protocol nodes and reusable modules.

### Stage 5: Generalize for other DML / pulp-app workflows

Use the same architecture for:

```text
Pulp diagnosis
Operative dentistry
Caries risk
Extractions
Trauma
Post-op complications
Recall / follow-up
```

## Recommended First Component

Start with this component:

```tsx
type DecisionCardProps = {
  node: ProtocolNode;
  caseData: EndoCase;
  onDecision: (option: DecisionOption) => void;
};

export function DecisionCard({ node, caseData, onDecision }: DecisionCardProps) {
  return (
    <section>
      <p>{node.phase}</p>
      <h2>{node.title}</h2>
      <p>{node.chairsideInstruction}</p>

      {node.safetyNotes?.length ? (
        <aside>
          {node.safetyNotes.map((note) => (
            <p key={note}>{note}</p>
          ))}
        </aside>
      ) : null}

      <div>
        {node.options.map((option) => (
          <button key={option.label} onClick={() => onDecision(option)}>
            {option.label}
          </button>
        ))}
      </div>
    </section>
  );
}
```

Build everything else around this.

Build the endodontic guide as the first example of a broader **clinical decision + note generation framework**.

The long-form endodontic protocol becomes the source data. The web app becomes the real chairside tool:

```text
Current clinical action
+

+
Safety warning
+
Structured measurements
+
Automatic note generation
```

This would fit naturally into the DML / pulp-app ecosystem and could later be reused for operative dentistry, pulp diagnosis, caries management, trauma, and post-op complication workflows.
